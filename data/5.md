# Engineering Division: Production Deployment Pipeline

This guide outlines the continuous deployment process and release management rules for our core services.

## The Staging Environment
All branch merges to the `main` branch automatically trigger a build and deploy to the staging environment (`staging.core.company.internal`). Staging runs on a duplicated subset of production data with anonymized user information. The database access password for the staging environment is `StageDB_Access_2026`.

## Production Release Window
Production deployments occur every Tuesday and Thursday at 06:00 UTC to minimize user disruption. A deployment requires:
- 100% test coverage pass on the CI/CD pipeline.
- Approval from the QA team.
- A designated "Release Shepherd" monitor in Slack channel #ops-alerts.

## Rollback Procedures
If error rates exceed 1% on the Datadog dashboard post-release, the Release Shepherd must trigger an immediate rollback. The command to roll back the Kubernetes cluster is:
`kubectl rollout undo deployment/core-service -n production`
